# Walmart Recruiting: Store Sales Forecasting

## კონკურსის მიმოხილვა

[Walmart Recruiting: Store Sales Forecasting](https://www.kaggle.com/competitions/walmart-recruiting-store-sales-forecasting) კონკურსის მიზანია Walmart-ის 45 მაღაზიის სხვადასხვა დეპარტამენტის კვირეული გაყიდვების პროგნოზირება ისტორიული მონაცემების საფუძველზე. ამოცანა ფასდება WMAE (Weighted Mean Absolute Error) მეტრიკით predicted და რეალურ გაყიდვებს შორის, სადაც სადღესასწაულო კვირებს 5-ჯერ მეტი წონა აქვთ. შესაბამისად, დღესასწაულების ზუსტი პროგნოზირება ამ ამოცანაში განსაკუთრებით მნიშვნელოვანია.

## მიდგომა
- დღესასწაულები: Super Bowl, Labor Day, Thanksgiving, Christmas.
- ტესტის rows-ების დაახლოებით 7.8% სადღესასწაულო კვირაა, ანუ ქულის დიდი ნაწილი სწორედ ამ კვირებზე იგება ან იკარგება.

![total weekly sales](images/total_weekly_sales.png)

EDA-ში კარგად ჩანს ამოცანის მთავარი სტრუქტურა: ძლიერი **წლიური სეზონურობა** (Thanksgiving და წინასაშობაო პიკები) და ის, რომ ეს პატერნი წლიდან წლამდე მეორდება:

![weekly seasonality by year](images/weekly_seasonality_by_year.png)

![autocorrelation by lag](images/autocorrelation_lags.png)

lag-52-ის (შარშანდელი იგივე კვირა) მედიანური ავტოკორელაცია ~0.70-ია, რაც ყველა ჩვენს მოდელში მთავარი სიგნალია: ხეებში feature-ად, DL მოდელებში input window-თი, კლასიკურ მოდელებში სეზონური წევრებით.


დავალების მიხედვით უნდა გამოგვეკვლია სხვადასხვა არქიტექტურის ოჯახები, ამიტომ ავიღეთ **8 მოდელი 4 ოჯახიდან**:

- **Tree-Based (გლობალური, tabular):** XGBoost, LightGBM
- **Deep Learning (გლობალური, Darts):** DLinear, N-BEATS, TFT (დავალება TFT-სა და PatchTST-დან ერთს ითხოვს; PatchTST Darts-ში არ არის, ამიტომ TFT ავირჩიეთ)
- **კლასიკური სტატისტიკური (ლოკალური, per-series):** ARIMA, SARIMA
- **Prophet (ლოკალური, per-series)**

ორივე ტიპის მოდელისთვის ერთი და იგივე შეფასების სისტემა გვაქვს, რომ არქიტექტურები სამართლიანად შევადაროთ:

- **საერთო ვალიდაცია** (`src/validation.py`): WMAE + 3 ფიქსირებული rolling-origin fold-ი. ორივე წევრი ზუსტად ერთსა და იმავე fold-ებს და მეტრიკებს ვიყენებთ.
- **საერთო MLflow სტრუქტურა**: ყველა არქიტექტურას აქვს ცალკე experiment-ი `X_Training`, შიგნით კი run-ები: `X_Preprocessing`, `X_Feature_Engineering`, `X_Feature_Selection`, `X_CV` (თითო კონფიგურაციაზე თითო nested run-ი, სადაც **ყველა ჰიპერპარამეტრი** ილოგება) და `X_Final` (საბოლოო pipeline + submission).
- **Pipeline მოთხოვნა**: ყველა არქიტექტურის საუკეთესო მოდელი შენახულია როგორც pyfunc pipeline, რომელიც პირდაპირ **raw test.csv-ზე** ეშვება (cleaning + reshaping + პროგნოზი ერთ ობიექტში).

გუნდური მუშაობა: EDA და საერთო ინფრასტრუქტურა (`src/`) ერთად გავაკეთეთ, შემდეგ მოდელები გავინაწილეთ და პარალელურად ვმუშაობდით, ყველა run საერთო DagsHub რეპოზიტორიაზე ილოგება.

## ვალიდაცია: საერთო 3-fold rolling-origin CV

ჩვეულებრივი random split დროით მწკრივებზე მომავლის ინფორმაციას გაჟონავდა, ამიტომ სამი ფიქსირებული, დროში მზარდი fold-ი გვაქვს:

| fold | train ბოლო | validation პერიოდი | რას ამოწმებს |
|------|-----------|--------------------|--------------|
| fold1_holiday | 2011-10-28 | 2011-11-04 .. 2012-01-27 | **Thanksgiving + Christmas**, leaderboard-ის მთავარი პროგნოზი |
| fold2_spring | 2012-02-03 | 2012-02-10 .. 2012-04-27 | არასეზონური პერიოდი |
| fold3_recent | 2012-07-27 | 2012-08-03 .. 2012-10-26 | ტესტთან ყველაზე ახლო პერიოდი |

ყველა მოდელში fold1 ყველაზე რთული აღმოჩნდა (სადღესასწაულო პიკები + 5x წონა), ამიტომ მოდელების შედარებისას მხოლოდ საშუალოს კი არა, fold1-საც ვუყურებდით.

## მონაცემთა დამუშავება / გაწმენდა (Data Preprocessing / Cleaning)

### საერთო cleaning (`Cleaner`, ყველა მოდელისთვის)

- `stores.csv` და `features.csv` ერთიანდება ძირითად ცხრილთან.
- **MarkDown1..5**: NaN ნიშნავს, რომ აქცია არ ყოფილა, ამიტომ 0-ით ივსება (და არა საშუალოთი).
- **CPI / Unemployment**: NaN მხოლოდ ტესტის ბოლო თვეებშია, ივსება per-store forward fill-ით (მხოლოდ წარსული მნიშვნელობებით, ანუ leakage არ არის).
- **უარყოფითი გაყიდვები** (დაბრუნებები, rows-ების 0.3%): ხეებისთვის ეს A/B ტესტით გადავწყვიტეთ (ქვემოთ), დანარჩენებში 0-ზე იჭრება.

ხეების Preprocessing ეტაპზე A/B-ით შევადარეთ negative clipping და log1p target-ი (default კონფიგურაციის მოდელით, 3-fold CV):

| კონფიგურაცია | XGBoost wmae_mean | LightGBM wmae_mean |
|--------------|-------------------|--------------------|
| **clip_neg + raw target (გამარჯვებული)** | **1,967.9** | **1,993.5** |
| keep_neg + raw target | 1,986.5 | 1,997.7 |
| clip_neg + log1p target | 2,058.5 | 2,051.4 |

log1p-მ ხეებს არ უშველა (ხეები ისედაც არაწრფივია), ხოლო negative-ების მოჭრამ ოდნავ მოუმატა. holiday sample_weight=5 ყველა run-ში ფიქსირებულია, რომ ტრენინგის loss-ი ზუსტად WMAE-ს ემთხვეოდეს.

### Time-Series reshaping (DL და კლასიკური მოდელებისთვის)

Darts-ის მოდელებს რეგულარული სიხშირის მწკრივები სჭირდებათ, ამიტომ ყველა Store x Dept წყვილი გადადის W-FRI კალენდარზე. ეს გადაწყვეტილებები ერთხელ დავაფიქსირეთ და ყველა TS მოდელის `Preprocessing` run-ში დავლოგეთ რეალური რაოდენობებით:

| გადაწყვეტილება | მნიშვნელობა | რაოდენობა |
|-----------------|-------------|-----------|
| შიდა გამოტოვებული კვირები | ივსება 0-ით (გამოტოვებული კვირა = გაყიდვა არ ყოფილა) | 27,667 კვირა |
| ადრე დამთავრებული მწკრივები | ბოლომდე ივსება 0-ით, რომ predict(n) ყველასთვის ერთ კალენდარზე დაჯდეს | 372 მწკრივი (12,816 კვირა) |
| გვიან დაწყებული მწკრივები | რჩება თავისი საწყისი თარიღით, უკან არ ივსება | 376 მწკრივი |
| სულ მწკრივი | | 3,331 |

- **Target ტრანსფორმაციები**: DLinear/N-BEATS: log1p + per-series minmax; TFT: მხოლოდ per-series minmax (quantile loss-ი მასშტაბის მიმართ მდგრადია); ARIMA/SARIMA: log1p; Prophet: raw (მულტიპლიკაციურობას თავად მოდელი ფარავს).

![target log1p](images/target_log1p.png)

- **Fallback**: მოკლე მწკრივები, fit-ჩავარდნები და cold-start წყვილები (ტესტში 11 წყვილი / 36 row-ა, რომლებიც train-ში საერთოდ არ არიან) სეზონურ-ნაივურ პროგნოზზე გადადიან (შარშანდელი იგივე კვირა, department-ის მედიანით შევსებული). ყველა Final run-ში ილოგება რამდენი row წავიდა fallback-ზე.
- **Loss-ის გასწორება WMAE-ზე**: DL მოდელები Darts-ის per-time-step `sample_weight`-ით ტრენინგდებიან (5x წონა სადღესასწაულო კვირებზე), ანუ L1 loss-ი ზუსტად WMAE ხდება, ისევე როგორც ხეებში sample_weight=5. კლასიკურ მოდელებს (Gaussian MLE / MAP fit) ეს შესაძლებლობა არ აქვთ და ეს მათი ერთ-ერთი სტრუქტურული მინუსია ამ მეტრიკაზე.

## Feature Engineering

### Tabular feature-ები ხეებისთვის (`feateng.py`)

ყველა history feature ტრენინგის მონაცემებში **lookup-ია** და არა shift, ამიტომ იგივე კოდი უშეცდომოდ მუშაობს train-ზე, ვალიდაციაზე და raw ტესტზე (leakage-ის გარეშე):

| ჯგუფი | feature-ები | დასაბუთება |
|-------|-------------|------------|
| lags | lag_52, lag_104 | წლიური სეზონურობა (EDA: r=0.70 და 0.54). მოკლე ლაგები 39-კვირიან ჰორიზონტზე ტესტის დროს არ არსებობს |
| lag_windows | roll_mean_52, roll_std_52 | lag-52-ის გარშემო მოძრავი ფანჯარა (სტაბილურობა) |
| calendar | Year, Month, Week | კალენდარული პოზიცია |
| holiday | days_to_thanksgiving, days_to_christmas, is_thanksgiving_week, is_pre_christmas_week, IsHoliday | EDA-ს ორი ხაფანგი: საშობაო პიკი 51/52 კვირებს შორის ცურავს და IsHoliday წინასაშობაო პიკის კვირას საერთოდ არ ნიშნავს |
| statics | Size, Type | მაღაზიის მახასიათებლები |
| markdowns / macro | MarkDown1..5, Temperature, Fuel_Price, CPI, Unemployment | side ცხრილებიდან |

### Future covariates DL მოდელებისთვის (`ts_data.py`)

`features.csv` მთელ სატესტო პერიოდს ფარავს (2013-07-26-მდე), ამიტომ ეს სვეტები Darts-ში **future covariates**-ად შეგვაქვს, ანუ მოდელმა პროგნოზის მომენტში მომავლის კალენდარი და აქციები "იცის":

- `calendar_holiday` preset: wk_sin/wk_cos და month_sin/month_cos (წლიური ციკლის Fourier წყვილები, რომ 52-ე და 1-ლი კვირა ერთმანეთის გვერდით აღიქმებოდეს), დღესასწაულებამდე დარჩენილი დღეები და ფლაგები.
- `full` preset: + markdowns, CPI, Temperature, Fuel_Price, Unemployment (გადარქმეული და minmax დასქეილებული).
- **სტატიკური კოვარიატები**: Store და Dept, როგორც მთელრიცხვა კოდები TFT-ის **categorical embedding**-ებისთვის (nominal ID-ები რიგითობის გარეშე ქსელში დასქეილებულ რიცხვებად არ უნდა შევიდეს), Type ორდინალურად, Size უწყვეტად.
- **Prophet**: Darts wrapper-ი თავად არქმევს `Date -> ds`, `Weekly_Sales -> y` და მოდელს გადაეცემა ჩვენი **holidays ცხრილი**: 4 სახელდებული დღესასწაული + **წინასაშობაო კვირა** (რომელსაც IsHoliday აცდენს), 2010-2013 წლების ზუსტი პარასკევის თარიღებით.

## Feature Selection

ეს ეტაპი არქიტექტურების მიხედვით ძალიან განსხვავებულია და თითოეულის ლოგიკა ცალკე run-შია დალოგილი:

### ხეები: gain + permutation importance

ორი სიგნალი (fold3-ის fit-იდან): gain (რამდენად გამოიყენა ხემ feature-ი) და permutation (რამდენად უარესდება ვალიდაციის WMAE სვეტის შემთხვევითი არევისას; თუ <= 0, feature-ში out-of-sample სიგნალი არ არის). კანდიდატი policy-ები 3-fold CV-თი გადამოწმდა:

| policy | XGBoost wmae_mean | LightGBM wmae_mean |
|--------|-------------------|--------------------|
| all_features | 1,966.9 | 2,026.6 |
| drop_low_gain | 1,947.4 | 2,026.6 |
| drop_perm_zero | 1,901.6 | 1,987.7 |
| **drop_union (გამარჯვებული)** | **1,901.0** | **1,987.7** |

- XGBoost-მა გადაყარა: IsHoliday, MarkDown3, MarkDown5, Month, Type (IsHoliday-ის გადაგდება ლოგიკურია: ჩვენი holiday distance feature-ები მასზე ბევრად ინფორმაციულია).
- LightGBM-მა გადაყარა: Fuel_Price, MarkDown1, MarkDown3, MarkDown5, Month, Week.

### დანარჩენი არქიტექტურები: არქიტექტურული სისუფთავის გადაწყვეტილებები

- **DLinear**: paper-native ვარიანტი **უნივარიატულია** (Zeng et al. 2022-ში ეგზოგენური ცვლადები საერთოდ არ არის; Darts-ის კოვარიატები ბიბლიოთეკის დანამატია). ჩვენ paper-ის ერთგულები დავრჩით: მხოლოდ target მწკრივი.
- **N-BEATS**: Darts-ის იმპლემენტაცია კოვარიატებს საერთოდ არ იღებს, ამიტომ სეზონურობა input window-დან უნდა ამოიკითხოს (ამიტომაა CV სივრცეში 52 და 65 კვირიანი input-ები).
- **TFT**: ერთადერთი კოვარიატებზე მშიერი არქიტექტურა; preset-ის არჩევანი (calendar_holiday vs full) CV-ს ჰიპერპარამეტრად დავუტოვეთ.
- **ARIMA / SARIMA**: მხოლოდ **სუფთა** მოდელები. ეგზოგენური რეგრესორები მოდელს ARIMAX/SARIMAX-ად აქცევდა, რაც ჩვენი გადაწყვეტილებით ამოცანის ფარგლებს გარეთაა. თეორიული პუნქტი: არასეზონურ ARIMA-ს 52-კვირიანი ციკლის დაჭერის მექანიზმი საერთოდ არ აქვს, SARIMA-ს კი აქვს (m=52), მაგრამ 143 კვირა მხოლოდ 2.75 წლიური ციკლია და D=1 სეზონური დიფერენცირების შემდეგ შესაფასებლად ძალიან ცოტა წერტილი რჩება.
- **Prophet**: trend + yearly seasonality + holidays ცხრილი; დამატებითი რეგრესორები არ გამოგვიყენებია.

## ტრენინგი და ექსპერიმენტები

ყველა მოდელი იზომება ერთი და იმავე მეტრიკებით: `wmae_fold1`, `wmae_fold2`, `wmae_fold3`, `wmae_mean`. CV ეტაპზე ხეებსა და DL მოდელებში Optuna TPE გამოვიყენეთ, კლასიკურ მოდელებში კი სრული grid-ი; ორივე შემთხვევაში თითო კონფიგურაცია ცალკე nested run-ია **ყველა** ჰიპერპარამეტრით.

### 1. XGBoost (30 Optuna trial)

საუკეთესო კონფიგურაცია: `n_estimators=1036, learning_rate=0.036, max_depth=11, min_child_weight=2.14, colsample_bytree=0.80, subsample=0.94, reg_alpha=0.50, reg_lambda=0.48`

| მეტრიკა | fold1 (holiday) | fold2 | fold3 | mean |
|---------|-----------------|-------|-------|------|
| wmae | 2,345.5 | 1,683.5 | 1,347.7 | **1,792.2** |

- train_wmae_mean = 926.4, ანუ gap_mean = 865.8. ხე საკმაოდ ღრმაა (depth=11) და ტრენინგზე ბევრად უკეთესია ვიდრე ვალიდაციაზე, თუმცა ვალიდაციის შედეგი მაინც საუკეთესოა ხეებს შორის, ანუ ეს "პროდუქტიული" overfit-ია და არა საზიანო.
- fold1 ყველაზე რთულია (2,345 vs 1,347 fold3-ზე): სადღესასწაულო კვირების 5x წონა ზუსტად იქ ვარდება, სადაც პროგნოზირება ისედაც რთულია.

### 2. LightGBM (30 Optuna trial)

საუკეთესო კონფიგურაცია: `n_estimators=1130, learning_rate=0.036, num_leaves=111, min_child_samples=43, colsample_bytree=0.87, subsample=0.87, reg_alpha=0.003, reg_lambda=0.52`

| მეტრიკა | fold1 (holiday) | fold2 | fold3 | mean |
|---------|-----------------|-------|-------|------|
| wmae | 2,563.1 | 1,732.1 | 1,399.6 | **1,898.3** |

- gap_mean = 327.9, ანუ LightGBM ბევრად ნაკლებად ოვერფიტდება ვიდრე XGBoost (865.8), მაგრამ ვალიდაციაზე მაინც ~106-ით ჩამორჩება. ორივე მოდელს თითქმის იდენტური learning_rate აურჩია Optuna-მ (~0.036), რაც კარგი ნიშანია, რომ search სივრცე გონივრულ ოპტიმუმს პოულობს.

### 3. DLinear (30 Optuna trial, უნივარიატული)

DLinear მწკრივს ორ ნაწილად შლის (moving-average trend + remainder) და თითოეულზე თითო წრფივ ფენას სწავლობს. საუკეთესო კონფიგურაცია: `input=65 კვირა, output=13 კვირა, kernel_size=25, const_init=True, batch=512, lr=0.00057`

| მეტრიკა | fold1 (holiday) | fold2 | fold3 | mean |
|---------|-----------------|-------|-------|------|
| wmae | 3,063.1 | 1,847.5 | 1,404.6 | **2,105.1** |

- 65-კვირიანი input-ის არჩევა ლოგიკურია: მოდელი სრულ წლიურ ციკლს + ბუფერს უნდა ხედავდეს, რომ სეზონურობა კოვარიატების გარეშე დაიჭიროს.
- fold3-ზე (1,404.6) თითქმის ხეების დონეზეა, fold1-ზე კი მკვეთრად ჩამორჩება: წრფივ დეკომპოზიციას სადღესასწაულო პიკების სიმკვეთრე უჭირს.

### 4. N-BEATS (25 Optuna trial, უნივარიატული)

საუკეთესო კონფიგურაცია generic არქიტექტურა აღმოჩნდა: `input=65, output=13, num_stacks=8, num_blocks=1, num_layers=2, layer_widths=512, dropout=0.1, batch=512, lr=0.00077`

| მეტრიკა | fold1 (holiday) | fold2 | fold3 | mean |
|---------|-----------------|-------|-------|------|
| wmae | 3,203.8 | 1,955.3 | 1,503.6 | **2,220.9** |

- საინტერესოა, რომ **generic** ვარიანტმა (თავისუფალი ბაზისები) აჯობა interpretable-ს (trend + seasonality სტეკები): როცა ბაზისებს შეზღუდვას ვუწესებთ, მოდელი უფრო ახსნადია, მაგრამ ამ მონაცემებზე ცოტა უფრო სუსტი.
- ორივე უნივარიატული ქსელი (DLinear, N-BEATS) ერთმანეთის მსგავს შედეგზეა და ორივე ხეებზე სუსტია: ხეებს lag-52 პირდაპირ მიეწოდებათ feature-ად + მთელი კოვარიატების ცხრილი, ქსელებმა კი იგივე ინფორმაცია მხოლოდ ისტორიის ფანჯრიდან უნდა ამოიღონ.

### 5. TFT (12 Optuna trial, კოვარიატებით და embedding-ებით)

პროექტის ყველაზე მძიმე მოდელი. საუკეთესო კონფიგურაცია: `input=52, output=13, hidden_size=32, lstm_layers=2, heads=4, full_attention=True, dropout=0.16, hidden_continuous_size=16, batch=256, lr=0.003, covariate_preset=calendar_holiday`

| მეტრიკა | fold1 (holiday) | fold2 | fold3 | mean |
|---------|-----------------|-------|-------|------|
| wmae | 4,631.4 | 2,879.0 | 2,277.2 | **3,262.5** |

- ტრენინგი quantile regression likelihood-ით მიდის (pinball loss q=0.5-ზე ზუსტად L1-ია) + 5x holiday sample weight.
- **მნიშვნელოვანი აღმოჩენა**: Darts-ში likelihood-იან მოდელზე `predict(num_samples=1)` მედიანას კი არ აბრუნებს, არამედ **ერთ შემთხვევით sample-ს იღებს**, ანუ პროგნოზი სტოქასტური გამოდიოდა. ეს ჩვენმა predict-twice parity ტესტმა დაიჭირა. გამოსავალი: pipeline-ი quantile **პარამეტრებს** აპროგნოზებს დეტერმინისტულად და q0.50 მედიანას იღებს, 39-კვირიან ჰორიზონტზე კი მედიანური გზის autoregressive rollout-ს აკეთებს.
- შედეგი მაინც ყველაზე სუსტია. მიზეზები გულწრფელად: მილიონობით პარამეტრიანი არქიტექტურა მხოლოდ 15 ეპოქითა და 12 trial-ით (კომპიუტის ბიუჯეტის გამო ლოკალურ Apple MPS-ზე გაშვებული), წყვილზე მხოლოდ ~90-140 კვირიანი ისტორია და 5-ზე ნაკლები სადღესასწაულო ეპიზოდი, ანუ TFT-ის ძალა (დიდი მონაცემები, მდიდარი კოვარიატები) აქ ვერ ვლინდება. საინტერესოა ისიც, რომ Optuna-მ **calendar_holiday** (მჭლე) preset-ი აურჩია full-ს, ანუ markdowns/macro სვეტები ქსელს არ დაეხმარა.

### 6. Prophet (24 Optuna trial sample-ზე + სრულმასშტაბიანი Final)

ლოკალური მოდელია (თითო წყვილზე თითო fit), ამიტომ CV 300-წყვილიან სტრატიფიცირებულ sample-ზე გავუშვით (Type x მოცულობის კვინტილი, seed და შემადგენლობა დალოგილია), გამარჯვებული კი Final ეტაპზე **ყველა** წყვილზე გადამოწმდა 3-ვე fold-ით, ანუ ქვემოთ მოყვანილი რიცხვები სრულმასშტაბიანია.

საუკეთესო კონფიგურაცია: `changepoint_prior_scale=0.032, seasonality_prior_scale=0.34, holidays_prior_scale=2.17, seasonality_mode=additive, yearly_seasonality=20, n_changepoints=25` (+ ფიქსირებული: weekly/daily seasonality off, uncertainty_samples=0, holidays ცხრილი)

| მეტრიკა | fold1 (holiday) | fold2 | fold3 | mean |
|---------|-----------------|-------|-------|------|
| wmae | 2,145.5 | 1,683.0 | 1,395.1 | **1,741.2** |

- **პროექტის სიურპრიზი**: Prophet-მა CV-ში ყველას აჯობა, XGBoost-ის ჩათვლით, და განსაკუთრებით fold1-ზე (2,145 vs XGBoost-ის 2,345). ახსნა ლოგიკურია: (1) თითო წყვილს საკუთარი მოდელი აქვს და არა გლობალურ მოდელში გაზიარებული პარამეტრები; (2) ჩვენი holidays ცხრილი (წინასაშობაო კვირის ჩათვლით) ზუსტად იმ კვირებს აძლევს ცალკე ეფექტს, სადაც WMAE 5x-ს ითვლის; (3) yearly_seasonality=20 საკმაოდ მოქნილი Fourier სეზონურობაა.
- საინტერესოა, რომ **additive** რეჟიმმა აჯობა multiplicative-ს.

### 7. ARIMA (სუფთა, 30 order-ის grid sample-ზე)

Grid: p, q = 0..3 და d = 0/1 კომბინაციები (გადაგდებულია გადაგვარებული (0,d,0)), 300-წყვილიან sample-ზე, 3-fold CV. sample-ის მასშტაბით საუკეთესო ორდერი **ARIMA(3,0,3)**:

| კონფიგურაცია (sample CV) | fold1 | fold2 | fold3 | mean |
|--------------------------|-------|-------|-------|------|
| p3_d0_q3 (გამარჯვებული) | 4,580.0 | 2,078.6 | 2,019.9 | **2,892.8** |
| p3_d0_q2 | 4,598.6 | 2,053.7 | 2,027.0 | 2,893.1 |
| p2_d0_q2 | 4,612.2 | 2,080.7 | 2,048.1 | 2,913.7 |

- შედეგი ზუსტად ისეთია, როგორსაც თეორია გვპირდება: fold1-ზე (დღესასწაულები) ორჯერ უარესია ვიდრე დანარჩენებზე, რადგან არასეზონურ ARIMA-ს წლიური ციკლის მექანიზმი უბრალოდ არ აქვს; d=0 ორდერების გამარჯვება კი log-სქეილზე მწკრივების შედარებით სტაციონარულობაზე მიუთითებს.
- Final ეტაპი (გამარჯვებულის სრულმასშტაბიანი გადამოწმება + pipeline + submission) Colab სესიის შეცდომის გამო ჩაიჭრა და თავიდან ეშვება; საბოლოო ცხრილში sample-ის რიცხვს დროებით ვწერთ.

### 8. SARIMA (სუფთა, m=52)

24 კონფიგურაციის grid-ი (4 (p,d,q) x 3 სეზონური (P,Q) x D=0/1, m=52) 200-წყვილიან sample-ზე. Preprocessing / FE / FS ეტაპები დალოგილია, CV ამ მომენტისთვის ჯერ სრულდება (m=52 fit-ები პროექტში ყველაზე ნელია, თითო კონფიგურაცია წუთებს იღებს sample-ზეც კი). ეს ზუსტად ის შემთხვევაა, რაზეც დავალება გვაფრთხილებდა: SARIMA-ს ღირებულება ძირითადად თეორიული გარჩევაა და არა ტრენინგზე დახარჯული დრო.

მთავარი დასკვნები:

1. **რანჟირება თითქმის მთლიანად fold1-ზე წყდება**: fold2/fold3-ზე ტოპ-5 მოდელი ერთმანეთს უახლოვდება (1,350-2,000 შუალედი), სადღესასწაულო fold-ზე კი განსხვავება ორმაგდება. ვინც დღესასწაულებს იჭერს, ის იგებს, ეს პირდაპირ WMAE-ს 5x წონიდან გამომდინარეობს.
2. **დღესასწაულების ექსპლიციტური ცოდნა > არქიტექტურის სირთულე**: Prophet-ს (holidays ცხრილი) და ხეებს (holiday distance feature-ები + 5x sample weight) დღესასწაულები პირდაპირ მიეწოდებათ და ისინი ლიდერობენ; უნივარიატულმა ქსელებმა იგივე ისტორიიდან უნდა ამოიცნონ და ჩამორჩებიან; TFT-ს კოვარიატები აქვს, მაგრამ კომპიუტის ბიუჯეტი არ ეყო.
3. **ლოკალური vs გლობალური**: Prophet-ის გამარჯვება აჩვენებს, რომ ამ ამოცანაზე per-series მოქნილობა ღირებულია, 143 კვირა საკმარისია მარტივი დეკომპოზიციური მოდელის კარგად მოსარგებად.
4. კლასიკური ARIMA პატიოსან სტრუქტურულ ჭერს აჩვენებს: სეზონურობის მექანიზმის გარეშე სადღესასწაულო fold-ზე კონკურენცია შეუძლებელია.

## საუკეთესო მოდელის შერჩევა და Kaggle

- ყველა არქიტექტურის საუკეთესო კონფიგურაცია `X_Final` run-შია დალოგილი როგორც pyfunc pipeline, რომელიც **raw test.csv-ზე** ეშვება (cleaning + reshaping + fallback + პროგნოზი).
- ჯამური ჩემპიონი (ამჟამად **Prophet**, wmae_mean=1,741.2 და საუკეთესო fold1) დარეგისტრირდება MLflow **Model Registry**-ში; `model_inference.ipynb` მას პირდაპირ Registry-დან ჩამოტვირთავს და submission-ს დააგენერირებს.
- submission ფაილები თითო მოდელისთვის `submissions/` საქაღალდეშია; kaggle-ის ქულებს ატვირთვის შემდეგ ზემოთა ცხრილში შევავსებთ.

ყველა run დალოგილია: [DagsHub-ზე](https://dagshub.com/ZukaCS/ML_FINAL_PROJECT)

თითოეული არქიტექტურის experiment-ში დაილოგა:

- `X_Preprocessing`: ფიქსირებული cleaning/reshaping გადაწყვეტილებები რეალური რაოდენობებით (შევსებული კვირები, padded მწკრივები, fallback პოლიტიკა).
- `X_Feature_Engineering`: რა სვეტები აიგო, რა გადაერქვა, კოვარიატების სრული ცხრილი artifact-ად.
- `X_Feature_Selection`: რომელი სვეტები მიეწოდება მოდელს და რატომ (ხეებში importance-ის policy-ების შედარების ცხრილით).
- `X_CV`: თითო კონფიგურაციაზე nested run-ი **ყველა** ჰიპერპარამეტრით (searched + fixed) და wmae მეტრიკებით; Optuna-ს optimization history და param importance გრაფიკები parent run-ზე.
- `X_Final`: არჩეული კონფიგურაცია, საბოლოო მეტრიკები, pyfunc pipeline და submission.
